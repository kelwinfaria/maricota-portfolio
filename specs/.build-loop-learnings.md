## 2026-06-18 — Migração p/ GitHub do Kelwin + bug de hidratação + carousel + olhinho

### Bug raiz do dia: `<script>` inline vs hidratação do React (App Router)
- **Sintoma:** contagem de produtos nas pills sumia, pills não clicavam e o carousel não navegava — os 3 ao mesmo tempo. Parecia "rota travando" mas era 100% client-side.
- **Diagnóstico (medir, não adivinhar):** `curl` na página ao vivo + `grep` mostrou que o HTML do servidor vinha SEM as contagens (`.n`=0) e com `#cdots` vazio → tudo dependia do script no cliente. O script executável era servido **cru e correto** (o `>` que aparecia era só o payload RSC `__next_f.push`, dado morto, não o script real).
- **Causa:** o `<script dangerouslySetInnerHTML>` rodava durante o parse do HTML, ANTES da hidratação. Mutava nós dos Server Components (`#filters`, `#cdots`, `#ctrack`) — adicionava contagens, dots, clones. Na hidratação o React 19 via divergência DOM↔vDOM, disparava *recoverable hydration error* e re-renderizava as subárvores do zero, apagando mutações E event listeners. O script só roda uma vez e nunca mais.
- **Correção:** mover TODA a lógica imperativa para um client component (`src/components/home/HomeScript.tsx`) dentro de `useEffect(() => {...}, [waUrl])`. Roda DEPOIS da hidratação; o React não toca mais nas subárvores estáticas (sem state → sem re-render), então mutações e listeners persistem. Cleanup idempotente via array de `cleanups` (remove listeners, clones, dots, spans, clearInterval) → seguro pra StrictMode/dev double-invoke.
- **Regra geral deste projeto:** NUNCA injetar `<script>` imperativo que muta nós de Server Components no App Router. Toda manipulação direta de DOM vai em client component + `useEffect`.

### Carousel — slide vazando ~30% do próximo
- **Causa:** `.cslide{flex:none;min-width:100%}`. Com `flex:none` o flex-basis é `auto` → resolve pela largura intrínseca da imagem; `min-width:100%` só põe piso, sem teto. Fotos grandes esticavam o slide além de 100% e o próximo vazava.
- **Correção:** `.cslide{flex:0 0 100%;width:100%;max-width:100%}` (basis fixo em 100% + teto). Também fixei `width:100%` no `.ctrack` para o `translateX(-100%)` bater exatamente um slide.
- **Regra:** slide de carousel sempre `flex:0 0 100%` + `max-width:100%`, nunca `flex:none`+`min-width`.

- Função `resolveWa()` no cliente para normalizar número ou link antes de montar a URL do WhatsApp.

### Padrões deste projeto
- Rotas API ficam em `src/app/api/` com handlers GET/POST/PUT/DELETE por arquivo.
- Upload de imagens usa `supabaseAdmin.storage.from('maricota')` com path `products/<timestamp>-<random>.<ext>`.
- `saveProd` no admin usa `fetch` diretamente (sem o helper `api()`) para o upload porque é FormData — precisa checar `r.ok` manualmente.
- Soft delete: coluna `deleted_at timestamptz` em `products`. Toda query pública DEVE filtrar `.is('deleted_at', null)`.
- Portfólio público (`page.tsx`) é Server Component com `force-dynamic` — busca dados direto do Supabase, sem passar pelas rotas API.
- WhatsApp: o campo `wa_number` na tabela `settings` aceita número puro (ex: `5537999999999`) ou link completo (`https://wa.me/...`). Normalizar sempre com: `wa.startsWith('http') ? wa : 'https://wa.me/' + wa.replace(/\D/g, '')`.
- Produtos têm campo `wa` individual (override do número global) que precisa ser passado como `data-wa` no HTML e tratado pelo lightbox JS.
- Toda interatividade da home vive em `src/components/home/HomeScript.tsx` (client component, `useEffect`). NÃO usar `<script>` inline.
